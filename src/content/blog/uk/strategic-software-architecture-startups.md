Більшість архітектурних катастроф стартапів трапляються не через некомпетентність інженерів. Вони трапляються тому, що команда приймала правильні рішення для неправильного етапу. Архітектура мікросервісів, що цілком мала б сенс для організації з 200 інженерами, стає організаційним податком, що знищує компанію з 12 людей. Моноліт, що добре служив на посівному етапі, стає причиною, чому ви не можете поставляти функції на стадії Series B.

Це поетапний фреймворк, побудований на досвіді роботи з понад 60 інженерними організаціями — від продуктових команд без доходу до компаній, що обробляють мільярди подій на день. Мета — не надати вам універсальну архітектуру. Мета — дати вам фреймворк для прийняття архітектурних рішень, що залишаються узгодженими з вашими поточними обмеженнями та наступним горизонтом.

## Основний принцип: архітектура служить організації

Перш ніж заглиблюватись у технічні деталі, основна теза, що визначатиме все подальше:

> **Ваша архітектура — це не технічний артефакт. Це соціальний контракт між вашою інженерною командою, швидкістю продукту та операційними можливостями. Оптимізуйте відповідно.**

Закон Конвея — це не рекомендація. Ваша система відображатиме комунікаційну структуру вашої організації, незалежно від того, плануєте ви це чи ні. Єдине питання — чи ви навмисні щодо цього.

## Етап 1: Посівний — модульний моноліт

На посівному етапі ваші основні обмеження:
- **Розмір команди**: 2–8 інженерів, часто фахівці широкого профілю
- **Основний ризик**: Не знайти відповідність продукту ринку достатньо швидко
- **Другорядний ризик**: Побудувати щось, що доведеться цілком викинути

Архітектура, що найкраще виживає на цьому етапі — **модульний моноліт** — один розгортуваний блок із чіткими внутрішніми межами модулів.

### Як виглядає модульний моноліт насправді

Поширена помилка — ставити знак рівності між «моноліт» та «великий клубок грязі». Добре структурований модульний моноліт має таку саму логічну розбивку, як і мікросервіси, без операційних накладних витрат.

```
src/
├── modules/
│   ├── billing/
│   │   ├── billing.service.ts
│   │   ├── billing.repository.ts
│   │   ├── billing.types.ts
│   │   └── billing.routes.ts
│   ├── users/
│   │   ├── users.service.ts
│   │   ├── users.repository.ts
│   │   ├── users.types.ts
│   │   └── users.routes.ts
│   ├── notifications/
│   │   ├── notifications.service.ts
│   │   ├── notifications.repository.ts
│   │   └── notifications.types.ts
│   └── analytics/
│       ├── analytics.service.ts
│       ├── analytics.repository.ts
│       └── analytics.types.ts
├── shared/
│   ├── database/
│   ├── middleware/
│   ├── errors/
│   └── config/
└── app.ts
```

Ключова дисципліна: **модулі спілкуються лише через їхній публічний сервісний інтерфейс, ніколи не через прямий доступ до бази даних іншого модуля.** Якщо вашому модулю `notifications` потрібні дані користувача, він викликає `users.service.getUser()` — він не робить JOIN таблиці `users` безпосередньо.

Ця дисципліна дозволяє пізніше виділити модуль в окремий сервіс без повного переписування.

### Стратегія баз даних на посівному етапі

Запустіть один екземпляр PostgreSQL. Не дозволяйте нікому переконати вас у використанні окремих баз даних для кожного модуля на цьому етапі. Операційні накладні витрати та складність міжмодульних запитів того не варті.

Що слід робити з першого дня:
- **Логічне розділення схем** за допомогою схем PostgreSQL (не просто плоский простір імен таблиць). Ваш модуль `users` володіє схемою `users`. `billing` — схемою `billing`.
- **Дотримуйтесь дисципліни зовнішніх ключів** — це змушує вас думати про власність даних зараз, коли це дешево.
- **Репліки для читання** раніше, ніж думаєте, що вони потрібні — вони коштують 30 доларів на місяць і врятують вас, коли ваші аналітичні запити почнуть знищувати затримку запису.

### Дизайн API для довговічності

Ваші рішення щодо зовнішнього API на посівному етапі обмежуватимуть вас роками. Кілька обов'язкових шаблонів:

**Версіонуйте з першого дня, навіть якщо у вас лише v1.**

```
/api/v1/users
/api/v1/billing/subscriptions
```

Ніколи `/api/users`. Вартість додавання `/v2/` пізніше є величезною. Вартість включення з самого початку — нульова.

**Проектуйте для споживачів, а не для вашої моделі даних.** Найпоширеніша помилка — будувати API, що відображає структуру вашої бази даних. Ваша кінцева точка `/users` не повинна розкривати внутрішню структуру таблиці `user_account`. Вона повинна розкривати те, що споживачі дійсно потребують.

**Послідовно використовуйте дизайн, орієнтований на ресурси.** Виберіть REST або GraphQL і дотримуйтесь вибору. Гібридні підходи на посівному етапі створюють плутанину, що посилюється при масштабуванні.

## Етап 2: Series A — модульний моноліт під тиском

На Series A ваша команда зросла (зазвичай 15–40 інженерів) і ваш моноліт починає показувати напругу. Ви розпізнаєте симптоми:
- Час збірки перевищує 5–8 хвилин
- Розгортання здаються ризикованими, оскільки все розгортається разом
- Дві команди постійно конфліктують через міграції бази даних
- Один повільний запит впливає на час відповіді всього додатку

Це не момент для «переходу на мікросервіси». Це момент для **зміцнення вашого модульного моноліту** та хірургічного виділення.

### Прапори функцій: передумова для всього

Перш ніж говорити про виділення мікросервісів, перш ніж говорити про шардинг бази даних, вам потрібні зрілі прапори функцій. Вони є фундаментом безпечного безперервного розгортання у масштабі.

```typescript
// A minimal, production-ready feature flag implementation
interface FeatureFlagConfig {
  enabled: boolean;
  rolloutPercentage?: number;
  allowlist?: string[];
  metadata?: Record<string, unknown>;
}

class FeatureFlagService {
  private flags: Map<string, FeatureFlagConfig>;

  isEnabled(flagKey: string, context: { userId: string; orgId?: string }): boolean {
    const flag = this.flags.get(flagKey);
    if (!flag || !flag.enabled) return false;

    // Allowlist check takes priority
    if (flag.allowlist?.includes(context.userId)) return true;
    if (flag.allowlist?.includes(context.orgId ?? '')) return true;

    // Percentage rollout via consistent hashing
    if (flag.rolloutPercentage !== undefined) {
      const hash = this.hashUserId(context.userId);
      return (hash % 100) < flag.rolloutPercentage;
    }

    return true;
  }

  private hashUserId(userId: string): number {
    // FNV-1a hash for consistent distribution
    let hash = 2166136261;
    for (let i = 0; i < userId.length; i++) {
      hash ^= userId.charCodeAt(i);
      hash = (hash * 16777619) >>> 0;
    }
    return hash;
  }
}
```

Прапори функцій дозволяють:
- Розгортати код без випуску функцій
- Запускати A/B-тести на інфраструктурних змінах (не лише UX)
- Виділяти сервіси за прапором та поступово направляти трафік
- Аварійні вимикачі для небезпечних функцій у виробництві

Це єдина можливість з найвищим важелем, яку ви можете вбудувати у платформну інфраструктуру перед масштабуванням команди.

### Коли виділяти мікросервіс

Сигнали, що модуль готовий до виділення:

1. **Незалежні вимоги до масштабування** — вашому модулю `video-processing` потрібні машини з 32 ядрами. Ваш модуль `user-auth` добре працює на 2 ядрах. Їх спільне виконання змушує вас провізіонувати найдорожчий варіант для всього.
2. **Незалежний ритм розгортання** — команда, що відповідає за модуль, розгортається 15 разів на день, тоді як решта моноліту — двічі на тиждень. Зв'язок створює гальмо.
3. **Відмінний операційний профіль** — модуль має принципово різні вимоги SLA (99,99% проти 99,9%), вимоги до мови або потреби в ізоляції відповідності.
4. **Команда відповідає за нього наскрізь** — є чітка, стабільна команда, що відповідає за домен. Межі сервісів без меж команд створюють пекло розподіленого моноліту.

Що НЕ є сигналом для виділення:
- «Мікросервіси є сучасними»
- Модуль великий (розмір не є критерієм — критерієм є зв'язок)
- Новий інженер хоче спробувати Go

### CI/CD як конкурентна перевага

На Series A ваш конвеєр розгортання — це не DevOps-господарство — це стратегічний актив. Компанії, що можуть розгортатися 50 разів на день, рухаються швидше за компанії, що розгортаються щотижня, крапка.

Цільові етапи конвеєра та часові бюджети:

| Етап | Цільовий час | Що робить |
|---|---|---|
| Lint + перевірка типів | < 60 с | Виловлює синтаксичні помилки, помилки типів |
| Юніт-тести | < 3 хв | Швидкий зворотний зв'язок щодо логіки |
| Інтеграційні тести | < 8 хв | Тести контрактів БД та API |
| Збірка + пакування | < 4 хв | Створення виробничого артефакту |
| Розгортання на staging | < 5 хв | Автоматизовані smoke-тести |
| Розгортання на production | < 3 хв | Blue/green або canary |

Разом: **менше 25 хвилин від коміту до production**. Кожна хвилина понад це — тертя, що накопичується у швидкісне гальмування по всій організації.

## Етап 3: Series B та далі — навмисна декомпозиція

На Series B+ у вас, вірогідно, є 60+ інженерів, кілька продуктових напрямків та реальна організаційна структура. Архітектурне питання зміщується від «як ми це будуємо» до «як ми тримаємо 8 команд, що поставляють незалежно».

### Вирівнювання топології команд

Найважливіше архітектурне рішення на цьому етапі не пов'язане з технологіями. Воно стосується проведення меж сервісів, що відповідають структурі вашої команди.

Використовуйте фреймворк **Team Topologies** як ваш орієнтир:
- **Команди, орієнтовані на потік**, відповідають за наскрізні зрізи продукту. Вони повинні відповідати за повні сервіси або групи сервісів із мінімальними зовнішніми залежностями.
- **Платформні команди** будують внутрішні можливості (спостережуваність, розгортання, інфраструктура даних), які команди, орієнтовані на потік, споживають як самообслуговування.
- **Команди, що допомагають**, є тимчасовими — вони підвищують кваліфікацію команд, орієнтованих на потік, а потім розпускаються.

Поширений режим відмов на цьому етапі: виділення мікросервісів, що не відображаються на межі команд, створюючи архітектуру, яка вимагає постійної міжкомандної координації для зміни однієї функції.

### Спостережуваність з першого дня (обов'язково)

Якщо ви винесете одне з цієї статті, нехай це буде: **інструментуйте вашу систему перш ніж вам знадобляться дані, а не після того, як щось зламається.**

Ваш стек спостережуваності повинен включати:
- **Структуроване журналювання** з послідовними полями (`service`, `trace_id`, `user_id`, `duration_ms`)
- **Розподілене трасування** (OpenTelemetry є стандартом — не ставте на пропрієтарне рішення)
- **RED-метрики** для кожного сервісу: Rate (швидкість), Errors (помилки), Duration (тривалість)
- **Бізнес-метрики**, що важливі для стейкхолдерів, а не лише для інженерів

```typescript
// Structured logging — do this from day one
const logger = createLogger({
  level: 'info',
  format: {
    service: process.env.SERVICE_NAME,
    version: process.env.APP_VERSION,
    environment: process.env.NODE_ENV,
  },
});

// Every request handler should emit structured context
app.use((req, res, next) => {
  req.log = logger.child({
    trace_id: req.headers['x-trace-id'] ?? generateTraceId(),
    user_id: req.user?.id,
    request_id: generateRequestId(),
  });
  next();
});
```

Вартість ретроспективного додавання цього до розподіленої системи є величезною. Вартість включення з самого початку — два дні платформної роботи.

## Технічний борг як інвестиція, а не провал

Переосмислення, що змінює те, як технічні керівники повинні думати про технічний борг:

**Технічний борг — це не провал дисципліни. Це фінансове рішення.**

Коли ви взяли на себе технічний борг на посівному етапі, пропускаючи тестове покриття заради швидшого відвантаження, ви зробили раціональний вибір: ви позичили у майбутнього часу інженерів, щоб купити теперішню швидкість. Як і фінансовий борг, питання не в тому, брати його чи ні — а в тому, чи є умови прийнятними та чи є у вас план його обслуговування.

Борг, що є **задокументованим, обмеженим і запланованим**, є прийнятним. Борг, що є **прихованим, необмеженим і зростаючим**, є екзистенційним.

Практичні підходи:
- **Ведіть явний реєстр технічного боргу** — відстежуваний список відомих статей боргу з розрахунковою вартістю обслуговування та вартістю погашення
- **Виділяйте 20% спринтової потужності** на обслуговування боргу як неповоротний бюджетний елемент
- **Ніколи не додавайте борг до критичних шляхів** — автентифікація, виставлення рахунків та безпека повинні дотримуватися більш суворих стандартів
- **Корелюйте борг з інцидентами** — якщо відомий борговий елемент спричинив виробничий інцидент, його пріоритет негайно зростає

Технічні керівники, що успішно проходять усі три етапи, мають одну спільну рису: вони ставляться до архітектури як до живого, контекстуального рішення, а не до одноразової вправи з проектування. Вони переглядають, рефакторують і — коли необхідно — перебудовують. Компанії, що зазнають невдачі, — це ті, що приймають рішення на посівному етапі та пристрасно захищають його на Series B.

Архітектура — це не про те, щоб бути правим. Це про те, щоб бути правим зараз, зберігаючи варіанти на потім.
